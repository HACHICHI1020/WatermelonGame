# 🍉 [게임 이름: 예 - Watermelon Puzzle] 
> **유니티 2D 물리 엔진 및 오브젝트 병합 로직 이해를 위한 실습 프로젝트**

## 🎯 프로젝트 소개 (Project Overview)
이 프로젝트는 게임 QA 직무를 준비하며, **유니티 2D 물리 엔진의 충돌 판정과 객체의 상태 변화(병합, 파괴) 로직**을 이해하기 위해 제작한 '수박 게임(Suika Game)' 모작입니다.
플레이어의 조작 상태에 따른 예외 처리, 다중 충돌 시 발생하는 버그 원인 파악, 그리고 정밀한 게임 오버 판정 방식을 직접 구현하며 **내부 구조의 취약점을 찾아내는 QA의 시야**를 넓히는 데 집중했습니다.

## 🔍 QA 관점에서의 핵심 학습 내용 (Key Learnings)

이 프로젝트를 직접 개발하며 다음과 같은 2D 엔진의 동작 원리와 검증 포인트를 학습했습니다.

**1. 2D 물리 충돌과 오브젝트 병합 시점의 예외 처리 (`OnCollisionEnter2D` & `Destroy`)**
* 동일한 레벨의 과일 두 개가 충돌할 때, 상위 과일을 생성(`CombineFruit`)하고 기존 과일 2개를 파괴(`Destroy`)하는 로직을 구현했습니다.
* **[QA Point]** 다중 충돌이 동시에 발생할 때, 파괴 예정인 객체를 참조하여 발생하는 `NullReferenceException`이나, 한 번에 과일이 2단계 이상 점프하는 중복 병합 버그의 발생 원리를 구조적으로 이해하게 되었습니다.

**2. 박스 캐스트를 활용한 정밀한 판정선 검증 (`Physics2D.BoxCast`)**
* 단순한 Trigger가 아닌, 붉은색 기즈모(Gizmos)로 가시화된 `BoxCast`를 활용하여 과일이 상단 데드라인을 넘었는지 실시간으로 감지하도록 설계했습니다.
* **[QA Point]** 물리 엔진의 연산 주기(FixedUpdate)와 화면 렌더링 프레임 간의 차이로 인해 충돌 판정이 씹히거나, 겹쳐진 오브젝트가 밀려 올라오며 발생하는 오작동을 테스트하는 기준을 세웠습니다.

**3. 사용자 입력 예외 상황 및 상태 통제 (`FruitControlState`)**
* `Ready`, `Drag`, `Drop` 세 가지 상태(Enum)를 두어, 과일이 낙하하는 도중에는 플레이어가 추가 조작이나 중복 투하를 할 수 없도록 입력(Input)을 통제했습니다.
* **[QA Point]** 유저가 마우스를 빠르게 연타하거나 화면 밖으로 드래그를 벗어나는 등(Edge Case), 입력 상태가 꼬일 때 게임 사이클이 정지하는 현상을 방지하는 로직의 중요성을 배웠습니다.

## 📂 주요 스크립트 구조 (Scripts Overview)
* `GameManager`: 과일의 드래그 & 드롭 입력 상태 제어, 객체 풀링(과일 리스트) 및 UI/점수 상태 관리
* `Fruit`: 개별 과일의 낙하 상태 제어 및 `OnCollisionEnter2D`를 통한 동일 등급 과일 병합/파괴 로직 처리
* `GameOver`: `Physics2D.BoxCast`를 지속적으로 쏘아올려 제한선(DeadLine)을 침범한 과일 객체 감지
